业务自助跑数平台建设方案

一、项目背景
当前，数据部已经基于 DolphinScheduler 搭建了一套完整的数据处理流程，整体链路包括：
    从业务平台抽数至数仓 PostgreSQL
    数据质量校验及预警
    数仓工作流加工处理
    处理完成后通知业务方在帆软平台查看结果
现阶段，该流程主要由数据部负责执行和维护，业务方在需要查看结果时依赖数据部完成跑数后进行通知。

随着业务使用频率提升和协同需求增加，当前模式存在明显问题：
    业务方需要依赖数据部手动触发跑数，效率较低
    数据部承担了大量重复性操作工作，资源投入较高
    跑数入口集中在数据部，业务侧缺少自主性
    一旦业务需求增加，数据部容易成为流程瓶颈
因此，需要在不改变现有底层处理链路的前提下，建设一套面向业务方的自助化能力，让业务方在授权范围内自主触发跑数，数据部主要负责平台维护、规则配置和异常处理。


二、项目定位
本项目定位为：业务自助跑数平台

定位说明：本项目不建设自主填报能力，产品边界明确聚焦于：
    业务方自助触发跑数
    查看跑数执行状态与结果
    异常时联系数据部负责人处理


三、建设目标
本项目建设目标分为三个层面。
1. 业务目标
让被授权的业务人员在完成前置业务动作后，可通过统一平台自主触发数据处理流程，减少对数据部人工操作的依赖。
2. 管理目标
将数据部角色从“日常执行者”转变为“平台维护者、规则配置者、异常处理者”，减少重复性执行工作。
3. 平台目标
在开放业务自助执行能力的同时，确保整体流程具备：
    权限可控
    配置可管理
    执行可追踪
    日志可审计
    风险可收敛


四、建设原则
为确保平台可落地、可维护、可控，本项目遵循以下原则：
1. 业务界面简化原则
业务方前台只展示其被授权的跑数按钮，执行状态及基础执行信息，不暴露底层技术实现。
2. 底层能力封装原则
DolphinScheduler API、PostgreSQL 存储过程、执行参数等均由平台统一封装，不直接对业务开放。
3. 配置驱动原则
各业务按钮对应的执行逻辑、参数模板、权限范围、通知规则等，由管理员在流程控制与管理平台统一配置。
4. 最小授权原则
仅被授权的人员可以看到并使用对应跑数按钮，不对全员开放。
5. 统一管控原则
前台展示、任务执行、日志记录、状态管理、权限控制、通知机制全部纳入统一平台管理。


五、总体方案
本项目总体采用以下模式：业务应用前台 + 流程控制与管理平台


六、方案架构说明
1. 业务应用前台
业务方可见的前台界面仅保留必要能力，重点突出“可执行”和“可查看”。
前台功能包括：
    ·展示业务方有权限看到的跑数按钮
    ·发起跑数执行
    ·查看当前执行状态
    ·查看最近执行结果
    ·查看前台业务按钮执行日志
    ·异常时查看提示并联系数据部负责人

前台展示原则：业务方前台只展示业务语言，不展示技术语言。
例如：
    展示“更新线上订单数据”
    展示“更新线下成本结果”
    展示“刷新报表数据”
    而不展示：
        DolphinScheduler 工作流名称
        存储过程名称
        技术参数
        数据库信息
        技术错误堆栈

前台新增日志要求：保留业务按钮执行日志，用于满足业务最小追踪诉求。
日志内容仅包括：
    执行人
    执行时间
该日志用于回答“谁点了按钮、什么时候点的”，不承载技术执行明细。

2. 流程控制与管理平台
这是本项目的核心能力平台，既承接前台请求，也承接后台配置与管理。
其核心职责包括：
（1）流程控制能力
    接收前台跑数请求
    校验当前用户权限
    判断任务对应执行方式
    组装执行参数
    调用 DS API 或 PG 存储过程
    跟踪执行过程
    回传执行结果
    输出统一日志
（2）后台管理能力
    配置业务按钮
    配置按钮对应的执行逻辑
    配置 DS 工作流或 PG 存储过程映射关系
    配置参数模板
    配置通知对象
    配置可见人员与执行人员
    配置执行限制策略
    维护日志与审计记录

在流程控制与管理平台中将DS / PG 执行引擎作为底层执行引擎，复用现有能力
DolphinScheduler 适用场景：
    多节点链路流程
    存在前后依赖关系的工作流
    需要状态编排和节点跟踪的任务
    现有成熟的抽数流程和数仓加工流程
PostgreSQL 存储过程适用场景：
    单点型处理逻辑
    参数相对固定的数据库处理过程
    在数据库内执行效率更高的逻辑
执行原则：无论底层采用 DS 还是 PG，业务方均无感知，全部通过流程控制与管理平台统一调度和管理。

前台面向业务方使用，流程控制与管理平台面向数据部管理员维护


七、核心运行机制
1. 业务使用流程
    管理员预先完成用户授权
    业务方登录平台
    平台展示其有权限看到的跑数按钮
    业务方点击按钮发起执行
    流程控制与管理平台完成权限校验及任务下发
    底层引擎执行相应流程
    业务方查看执行状态与结果
    若失败，则根据提示联系数据部负责人处理
2. 管理员配置流程
    在流程控制与管理平台新增一个业务跑数任务
    配置任务展示名称
    配置任务说明与适用场景
    配置底层执行方式
    DS API
    PG 存储过程
    配置执行对象及参数模板
    配置可见人员
    配置执行人员
    配置通知策略
    发布后，相关业务人员可在前台看到对应按钮


八、权限设计
本项目权限不对全员开放，而是采用指定人员可见机制。
1. 权限原则
未被授权人员，前台不展示对应按钮
已被授权人员，才能看到并执行对应任务
管理员负责维护授权范围


九、日志设计
1. 前台业务按钮执行日志
用于满足业务最基础的追踪诉求。
日志内容：
    执行人
    执行时间
日志特点：
    面向业务方展示
    信息简洁
    不展示技术细节
    用于回答“谁在什么时候发起了这次执行”

2. 流程控制与管理平台日志
用于平台管理、问题排查和审计追踪。
建议记录内容：
    任务名称
    执行方式（DS / PG）
    发起人
    发起时间
    参数模板版本
    实际执行实例编号
    执行状态
    成功 / 失败结果
    错误摘要
    配置变更记录
    权限变更记录
    通知发送记录
日志用途：
    平台运维
    问题排查
    配置审计
    责任追踪


十一、风险评估
风险一：业务方误触发执行
虽然前台只保留按钮，降低了理解成本，但也可能导致误点、重复执行等问题。
应对措施：
    增加执行确认
    增加“执行中不可重复点击”机制
    增加重复触发拦截
    增加时间窗口控制

风险二：管理员配置错误导致任务执行异常
由于底层执行逻辑和参数全部在流程控制与管理平台中配置，配置错误会直接影响业务使用。
典型表现：
    配错工作流
    配错存储过程
    配错参数模板
    配错授权人员
    配错通知对象
应对措施：
    增加配置校验机制
    关键配置增加二次确认
    配置变更留痕

风险三：业务方不知道什么场景下该执行按钮
由于前台被简化为按钮式操作，如果按钮语义不足，业务方容易理解偏差。
应对措施：每个业务按钮建议补充最少信息
    任务名称
    适用场景
    执行前提
    执行影响范围
    预计耗时
    失败联系人

风险四：权限管理不及时导致可见范围混乱
采用“管理添加相关人员才可见”的机制后，人员新增、离岗、转岗都会影响权限准确性。
应对措施：
    权限绑定业务角色或组织
    建立定期权限复核机制
    人员变动时同步回收权限

风险五：DS 和 PG 双模式并行带来维护复杂度
一个平台同时支持 DS 工作流和 PG 存储过程，会增加平台统一抽象和运维难度。
应对措施：在流程控制与管理平台中统一抽象“任务模型”，不管底层如何执行，统一管理以下字段
    任务名称
    执行方式
    发起人
    发起时间
    执行状态
    结果摘要
    日志信息
