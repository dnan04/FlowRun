# 平台数据迁移指南

本文只迁移平台自身数据，不迁移业务库、PG 存储过程执行库、DolphinScheduler 自身数据库，也不迁移外部人员视图所在数据库。

当前平台后端通过 `backend/.env` 中的 `PLATFORM_DB_*` 和 schema 配置连接平台库：

```env
PLATFORM_DB_HOST=127.0.0.1
PLATFORM_DB_PORT=5432
PLATFORM_DB_DATABASE=postgres
PLATFORM_DB_USERNAME=postgres
PLATFORM_DB_PASSWORD=postgres
PLATFORM_DB_SSLMODE=prefer
PLATFORM_CONFIG_SCHEMA=dr_config
PLATFORM_RECORD_SCHEMA=dr_record
```

后端启动时会执行：

- 创建 `PLATFORM_CONFIG_SCHEMA`、`PLATFORM_RECORD_SCHEMA`。
- 通过 SQLAlchemy 创建当前模型中缺失的表。
- 通过运行时兼容逻辑补齐旧表缺失字段和部分索引。
- 写入默认目录、DS 下拉选项、默认管理员和系统初始化审计记录。

因此迁移时建议直接迁移这两个 schema 的结构和数据，保持 schema 名称不变最稳妥。

## 1. 迁移范围

默认 `dr_config` schema 中包含平台配置类数据：

- `sys_admin_user`：平台管理员账号。
- `dr_task_directory`：任务目录，支持父子目录。
- `dr_task_definition`：任务配置、发布状态、任务中心展示状态、执行窗口、测试结果、DS/PG 后置方法等。
- `dr_task_visible_user`：任务可见人员关系，保存外部人员 `email` 和姓名快照。
- `dr_task_execute_user`：任务可执行人员关系，保存外部人员 `email` 和姓名快照。
- `dr_task_notify_user`：任务失败联系人关系，保存外部人员 `email` 和姓名快照。
- `ds_project`：DolphinScheduler 项目下拉选项，前端展示 `name`，调用时使用 `code`。
- `ds_environment`：DolphinScheduler 环境下拉选项，前端展示 `name`，调用时使用 `code`。
- `ds_alertgroup`：DolphinScheduler 告警组下拉选项，前端展示 `group_name`，调用时使用 `id`。

默认 `dr_record` schema 中包含平台记录类数据：

- `dr_task_execution`：任务执行记录，包括请求人、执行参数、状态、外部实例号、结果摘要和错误摘要。
- `dr_audit_log`：审计日志，包括管理员操作和普通用户操作记录。

如果后续代码新增表，只要新表仍在 `PLATFORM_CONFIG_SCHEMA` 或 `PLATFORM_RECORD_SCHEMA` 下，使用本文的 schema 级迁移命令会一并迁移。

## 2. 迁移前准备

1. 停止旧环境后端服务，避免导出过程中继续写入新数据。
2. 确认旧环境 `backend/.env` 中的 `PLATFORM_DB_*`、`PLATFORM_CONFIG_SCHEMA`、`PLATFORM_RECORD_SCHEMA`。
3. 确认新环境 PostgreSQL 已创建目标数据库，并且目标用户有建 schema、建表、建序列、建索引和写入数据的权限。
4. 如果新环境曾经启动过后端，目标库中可能已经存在默认种子数据。恢复时建议使用 `--clean --if-exists` 覆盖同名对象。

## 3. 从旧 PostgreSQL 导出平台数据

如果旧环境使用默认 schema `dr_config` 和 `dr_record`，在 PowerShell 中执行：

```powershell
pg_dump -h 127.0.0.1 -p 5432 -U postgres -d postgres `
  -n dr_config -n dr_record `
  -F c -f E:\VSCodeProject\DateRunnerProject_pyv\platform_data_backup.dump
```

如果旧环境 schema 名称不是默认值，把命令中的 `dr_config`、`dr_record` 换成旧环境实际的：

- `PLATFORM_CONFIG_SCHEMA`
- `PLATFORM_RECORD_SCHEMA`

说明：

- `-n dr_config -n dr_record` 表示只导出平台使用的两个 schema。
- `-F c` 表示导出为 PostgreSQL custom 格式，便于后续使用 `pg_restore` 恢复。
- 导出文件示例路径为 `E:\VSCodeProject\DateRunnerProject_pyv\platform_data_backup.dump`，可按实际情况调整。
- 执行时如果提示输入密码，输入旧环境 `PLATFORM_DB_PASSWORD`。

## 4. 恢复到新 PostgreSQL

假设新环境平台库配置如下：

```env
PLATFORM_DB_HOST=新HOST
PLATFORM_DB_PORT=5432
PLATFORM_DB_DATABASE=新数据库名
PLATFORM_DB_USERNAME=新用户名
PLATFORM_DB_PASSWORD=新密码
```

在 PowerShell 中执行：

```powershell
pg_restore -h 新HOST -p 5432 -U 新用户名 -d 新数据库名 `
  --clean --if-exists `
  E:\VSCodeProject\DateRunnerProject_pyv\platform_data_backup.dump
```

说明：

- `--clean --if-exists` 会在恢复前删除目标库中同名对象，适合覆盖新环境已初始化出的空表和默认数据。
- 备份中包含原 schema 名称，恢复后 schema 名称会保持导出时的名称。
- 如果目标库是空库，也可以正常恢复。
- 执行时如果提示输入密码，输入新环境 `PLATFORM_DB_PASSWORD`。

## 5. 修改新环境 `backend/.env`

恢复完成后，把新环境 `backend/.env` 中的平台库配置改为新 PostgreSQL 地址。

示例：

```env
PLATFORM_DB_HOST=新HOST
PLATFORM_DB_PORT=5432
PLATFORM_DB_DATABASE=新数据库名
PLATFORM_DB_USERNAME=新用户名
PLATFORM_DB_PASSWORD=新密码
PLATFORM_DB_SSLMODE=prefer
PLATFORM_CONFIG_SCHEMA=dr_config
PLATFORM_RECORD_SCHEMA=dr_record
```

如果恢复后仍使用 `dr_config` 和 `dr_record`，schema 配置不用改。

如果恢复时保留的是旧环境的自定义 schema 名称，新环境 `.env` 也必须配置为相同的 schema 名称。

## 6. 外部依赖重新配置

以下数据源不属于平台数据迁移范围，需要在新环境 `.env` 中重新配置连接信息：

- 外部人员视图：`REPORT_USER_DB_*`、`REPORT_USER_VIEW_NAME`、`REPORT_USER_*_COLUMN`。
- PG 存储过程执行库：`PG_EXECUTION_*`。
- DolphinScheduler API：`DOLPHINSCHEDULER_*`。
- 前端访问后端的地址和 CORS：按部署方式检查 `VITE_API_BASE_URL`、`CORS_ORIGINS` 等配置。

平台表里只保存任务配置、授权 email、执行记录、DS 下拉选项等平台数据，不包含外部系统数据库本体。

## 7. 启动后检查

启动新环境后端后，后端会再次执行建表和兼容补字段逻辑。完成后检查：

- `/health` 返回正常。
- 管理员账号可以登录管理后台。
- 管理后台的任务配置、目录、发布状态、测试结果仍在。
- 任务中心目录和任务卡片展示正常。
- 任务可见人员、可执行人员、失败联系人 email 和姓名快照仍在。
- DS 项目、环境、告警组下拉选项展示正常。
- 执行记录和审计日志仍在。
- 普通用户可通过外部人员视图识别身份，并看到对应授权任务。
- DS 任务和 PG 任务在新环境连接配置正确后可正常执行。

## 8. 注意事项

- 只迁移平台数据时，不要做整库导出，避免把业务库、DolphinScheduler 库或外部人员库混入。
- 推荐继续使用原 schema 名称，例如 `dr_config`、`dr_record`，这样恢复后只需要修改数据库连接地址。
- 如果必须改 schema 名称，建议先恢复到原 schema，再用 PostgreSQL 的 schema rename 或重新导出 plain SQL 后谨慎替换 schema 名称；同时同步修改 `.env`。
- 新环境第一次启动前恢复数据，可以避免默认种子数据被创建后再覆盖。
- 新环境已经启动过也没关系，恢复时使用 `--clean --if-exists` 覆盖同名平台对象即可。
- `Base.metadata.create_all` 不会删除旧字段，运行时兼容逻辑只补齐当前代码需要的部分字段；完整迁移仍应以 `pg_dump` 导出的两个 schema 为准。
